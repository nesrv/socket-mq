# Socket-MQ Chat — учебный проект 

Лабораторный **текстовый чат** в браузере: **FastAPI**, **WebSocket**, **HTMX** (`htmx-ext-ws`), **RabbitMQ**, **PostgreSQL** и отдельный процесс **worker**. Цель — освоить realtime на практике и отличить вариант «всё в памяти одного сервера» от связки с брокером и базой.

Подробные пошаговые инструкции, листинги кода и сценарии сдачи — в файле `**laba-simple.md`**. Конспект лекции по системному дизайну — в `**lect.md**`.

## Что вы изучаете на примере проекта


| Идея                                   | Где проявляется                                                            |
| -------------------------------------- | -------------------------------------------------------------------------- |
| WebSocket для дуплекса клиент ↔ сервер | Маршруты `/ws/{room_id}`, HTMX `ws-send`                                   |
| Память процесса vs общее состояние     | Вариант без брокера (каталог `PART-0`) vs текущий корень репозитория       |
| Очередь сообщений                      | RabbitMQ, два ключа маршрутизации: создано → сохранено                     |
| Вынос тяжёлой работы из web            | `worker.py` пишет в PostgreSQL и снова публикует в очередь                 |
| Наблюдаемость и тесты                  | `pytest`, при желании `tests/ws_check.py`, CI в `.github/workflows/ci.yml` |

## Структура репозитория (кратко)

```
socket-mq/
├── app/                 # Приложение FastAPI (web)
│   ├── main.py          # HTTP, WebSocket, lifespan, связь с MQ
│   ├── db.py            # SQLAlchemy + asyncpg, модель сообщений
│   ├── mq.py            # aio-pika, exchange, очереди
│   ├── ws.py            # Менеджер комнат WebSocket
│   ├── templates/       # Jinja2 + HTMX
│   └── static/          # Стили
├── worker.py            # Consumer RabbitMQ → INSERT в БД → publish persisted
├── tests/               # test_health.py, ws_check.py
├── docker-compose.yml   # web, worker, postgres, rabbitmq
├── Dockerfile
├── requirements.txt
├── pytest.ini
├── .env.example         # Образец переменных окружения
├── laba-simple.md       # Методичка по лабораторной работе
├── lect.md              # Материалы к лекции
├── PART-0/              # Упрощённый вариант: только web, без RabbitMQ и Postgres
└── PART-1/              # Копия полного варианта (можно использовать как эталон)
```

Текущая **рабочая** конфигурация «часть 2» обычно совпадает с **корнем** репозитория; **PART-0** — как **вариант 0** из методички.

## Требования

- **Docker** и **Docker Compose** (plugin `docker compose`).
- **Порты на хосте:** `8100` (приложение), `5433` (Postgres снаружи), `5672` и `15672` (RabbitMQ и веб-интерфейс управления). Если заняты — см. комментарии в `laba-simple.md` про смену портов.

## Быстрый старт (полный стек)

1. Клонируйте репозиторий и перейдите в каталог проекта:

```bash
git clone <URL-репозитория>
cd socket-mq
```

1. Создайте файл `.env` из примера:

```bash
cp .env.example .env
```

При необходимости отредактируйте `.env` так, чтобы он согласовывался с `docker-compose.yml` (логин/пароль Postgres, пароль гостя RabbitMQ и т.д.).

1. Поднимите сервисы:

```bash
docker compose up -d --build
```

1. Проверьте работу:

- В браузере: [http://127.0.0.1:8100/health](http://127.0.0.1:8100/health) — ожидается JSON `{"status":"ok"}`.
- Страница комнаты: [http://127.0.0.1:8100/rooms/room-1](http://127.0.0.1:8100/rooms/room-1).
- Две вкладки с одной и той же комнатой: сообщение из одной вкладки должно появляться во второй (при запущенных **web**, **worker**, **rabbitmq**, **postgres**).

Панель RabbitMQ (логин/пароль по умолчанию у образа `management` — `guest` / `guest`): [http://127.0.0.1:15672](http://127.0.0.1:15672).

## Сервисы в `docker-compose.yml`


| Сервис       | Назначение                                                                   |
| ------------ | ---------------------------------------------------------------------------- |
| **web**      | Uvicorn + FastAPI, страницы, WebSocket, публикация в RabbitMQ                |
| **worker**   | Читает очередь входящих сообщений, пишет в PostgreSQL, публикует «сохранено» |
| **postgres** | База `chat`, таблица сообщений                                               |
| **rabbitmq** | Брокер AMQP + UI на порту **15672**                                          |


С хоста PostgreSQL доступен на `**localhost:5433`** (внутри Docker-сети приложение подключается к хосту `**postgres:5432**` — см. `DATABASE_URL` в `.env.example`).

## Тесты

Из **корня репозитория** (рядом с `pytest.ini`):

```bash
pip install -r requirements.txt
pytest -q
```

Или в контейнере **web**:

```bash
docker compose run --rm web pytest -q
```

- `**tests/test_health.py**` — проверка `GET /health` без поднятых Postgres/RabbitMQ (используются подмены в тесте).
- `**tests/ws_check.py**` — не pytest: ручной интеграционный скрипт при **полностью** запущенном `docker compose` и проброшенном порте **8100**:

```bash
python tests/ws_check.py
```

Подробнее — раздел **6** в `laba-simple.md`.
